# C++/Qt 项目文档读取 SKILL

**核心原则**：_ai_docs/ 文档是源码的超集——它包含源码的全部内容，
加上逐行分析、设计意图、线程安全分析、调用链图、已知问题等。
**读文档比读源码更快、信息更丰富，永远优先读文档。**

---

## 工作流总览

```
阶段 0：检测文档存在   → 确认 _ai_docs/ 目录存在并列出文档清单
阶段 1：加载入口文档   → 必读 00_AI_CONTEXT.md，建立项目全局认知
阶段 2：按需精确加载   → 根据用户问题，精确加载对应的专项文档
阶段 3：合成与回答     → 基于文档内容给出完整、精确的回答
阶段 4：主动补充       → 发现文档未覆盖的内容时，提示用户
```

---

## 阶段 0：检测文档

用户提供项目路径后，立即执行：

```bash
# ① 检测文档目录
ls -la {PROJECT_ROOT}/_ai_docs/ 2>/dev/null || echo "⚠ 未找到 _ai_docs/ 目录"

# ② 列出所有文档及大小
ls -lh {PROJECT_ROOT}/_ai_docs/*.md 2>/dev/null

# ③ 统计文档规模
wc -l {PROJECT_ROOT}/_ai_docs/*.md 2>/dev/null | tail -1
```

### 文档不存在时的处理

若 `_ai_docs/` 不存在或为空：
1. 告知用户：「该项目尚未生成 AI 文档，建议先运行 cpp-project-doc skill 生成文档」
2. 询问用户是否希望现在生成
3. 若用户坚持直接回答，则改用源码扫描（回退到 cpp-project-doc 的阶段0流程）

### 文档版本检查

```bash
# 检查文档生成时间（判断是否过期）
stat {PROJECT_ROOT}/_ai_docs/00_AI_CONTEXT.md 2>/dev/null | grep -E "Modify|Change"

# 检查源码是否比文档更新（若有则提示用户文档可能已过期）
find {PROJECT_ROOT} \( -name "*.cpp" -o -name "*.h" \) \
  | grep -v _ai_docs \
  | xargs ls -t 2>/dev/null | head -5
```

若发现源码文件比文档更新：
> ⚠ 注意：部分源码文件比文档更新，文档可能未反映最新修改。  
> 建议重新运行文档生成 skill 以获取最新状态。我将基于现有文档回答，  
> 并在涉及可能已变更的部分时加以说明。

## 阶段 1：加载入口文档（必须执行）

**无论用户问什么，第一步永远是读 `00_AI_CONTEXT.md`。**

```bash
cat {PROJECT_ROOT}/_ai_docs/00_AI_CONTEXT.md
```

读完后，在内部建立以下认知索引（不需要输出给用户）：

```
项目名称：_______________
核心功能：_______________
主要类列表：_____________（快速映射 类名 → 文件路径）
最重要的N个函数：_________
文档索引表：_____________（哪类问题去看哪个文档）
```

## 阶段 2：按需精确加载（核心策略）

根据用户问题类型，**只加载必要的文档**，避免无效读取。

### 2.1 问题 → 文档映射表

| 用户问题类型 | 应加载的文档 | 加载优先级 |
|------------|------------|-----------|
| "这个类做什么" / "成员变量有哪些" | `05_CLASS_REFERENCE.md` | 高 |
| "这个函数怎么实现的" / "逐行解释" | `06_FUNCTION_REFERENCE.md` | 高 |
| "用户点击 xxx 后发生了什么" | `07_DATA_FLOW.md` | 高 |
| "信号 xxx 连接到哪里" / "槽函数" | `08_QT_SIGNALS_SLOTS.md` | 高 |
| "控件 xxx 叫什么名字" / "UI布局" | `09_QT_UI_WIDGETS.md` | 高 |
| "xxx 什么时候被释放" / "内存泄漏" | `10_MEMORY_LIFECYCLE.md` | 高 |
| "线程安全吗" / "并发问题" | `11_THREADING.md` | 高 |
| "有什么 Bug" / "技术债" | `12_KNOWN_ISSUES.md` | 高 |
| "项目结构" / "模块划分" | `02_ARCHITECTURE.md` | 中 |
| "怎么编译" / "CMake 怎么写的" | `03_BUILD_SYSTEM.md` | 中 |
| "全局变量" / "宏定义" | `04_GLOBAL_SYMBOLS.md` | 中 |
| "项目是做什么的" / 综合性问题 | `01_PROJECT_OVERVIEW.md` | 中 |

### 2.2 加载策略

**策略A：单点定位**（最常用）

用户问具体类/函数/控件时，直接定位到对应文档的对应段落：

```bash
# 在类参考文档中精确定位某个类
grep -n "### 类：\`ClassName\`\|^## ClassName" \
  {PROJECT_ROOT}/_ai_docs/05_CLASS_REFERENCE.md

# 找到行号后，读取该类的完整描述（假设从第N行到第M行）
sed -n '{N},{M}p' {PROJECT_ROOT}/_ai_docs/05_CLASS_REFERENCE.md

# 在函数参考中定位某个函数
grep -n "函数：ClassName::methodName\|^### ClassName::methodName" \
  {PROJECT_ROOT}/_ai_docs/06_FUNCTION_REFERENCE.md
```

**策略B：全文加载**（适用于架构、综合性问题）

```bash
cat {PROJECT_ROOT}/_ai_docs/02_ARCHITECTURE.md
cat {PROJECT_ROOT}/_ai_docs/07_DATA_FLOW.md
```

**策略C：跨文档联动**（适用于复杂问题）

例如"解释信号 dataReady 的完整生命周期"：
1. 先在 `08_QT_SIGNALS_SLOTS.md` 找信号定义和连接
2. 再在 `06_FUNCTION_REFERENCE.md` 找 emit 位置的上下文
3. 再在 `07_DATA_FLOW.md` 找该信号所在的完整数据流场景

```bash
# 跨文档搜索某个符号
grep -n "dataReady" \
  {PROJECT_ROOT}/_ai_docs/08_QT_SIGNALS_SLOTS.md \
  {PROJECT_ROOT}/_ai_docs/06_FUNCTION_REFERENCE.md \
  {PROJECT_ROOT}/_ai_docs/07_DATA_FLOW.md
```

### 2.3 大型文档的分段读取

`05_CLASS_REFERENCE.md` 和 `06_FUNCTION_REFERENCE.md` 可能非常大（数千行）。
**禁止全量加载，必须精确定位**：

```bash
# 先获取文档的章节目录
grep -n "^### \|^## " {PROJECT_ROOT}/_ai_docs/05_CLASS_REFERENCE.md | head -50

# 确认目标类的行范围
START=$(grep -n "### 类：\`TargetClass\`" \
  {PROJECT_ROOT}/_ai_docs/05_CLASS_REFERENCE.md | cut -d: -f1)
END=$(awk "NR>$START && /^### 类：/{print NR; exit}" \
  {PROJECT_ROOT}/_ai_docs/05_CLASS_REFERENCE.md)

# 只读取该类的内容
sed -n "${START},${END}p" {PROJECT_ROOT}/_ai_docs/05_CLASS_REFERENCE.md
```

## 阶段 3：合成与回答规范

### 3.1 回答质量标准

基于文档回答时，必须满足：

**精确性**
- 变量名、函数签名、行号引用必须与文档一致
- 不得在文档明确说明的内容上进行"推测"
- 若文档中存在 ⚠ 警告，必须在回答中提及

**完整性**
- 涉及函数时：职责 + 关键参数 + 返回值 + 副作用 + 调用时机
- 涉及类时：用途 + 关键成员 + 设计模式 + 线程安全性
- 涉及数据流时：完整的调用链 + 线程切换点 + 数据变换过程

**关联性**
- 回答类的问题时，主动引用其信号槽连接（来自 08 文档）
- 回答函数的问题时，主动提及其在数据流中的位置（来自 07 文档）
- 发现相关的已知问题时，在回答末尾附上（来自 12 文档）

### 3.2 标准回答模板

**问：某个类是做什么的？**
```
## ClassName

**职责**：[来自 05_CLASS_REFERENCE.md 的类描述]

**继承关系**：[基类列表及每个基类的作用]

**设计模式**：[单例/工厂/观察者等]

**关键成员变量**：
| 变量名 | 类型 | 用途 |
...（只列最重要的3-5个）

**核心方法**：
- `method1()`：[一句话描述]
- `method2()`：[一句话描述]

**信号槽摘要**：
- 发出信号：[列出，来自 08 文档]
- 响应信号：[列出，来自 08 文档]

**线程安全**：[来自 11 文档或类文档中的线程安全标注]

⚠ 已知问题：[如有，来自 12 文档]
```

**问：某个函数是怎么实现的？**
```
## ClassName::methodName()

**完整签名**：[来自 06_FUNCTION_REFERENCE.md]

**调用时机**：[谁调用、什么场景触发]

**逐段逻辑**：
[直接引用文档中的逐行/逐段分析，保留行号引用]

**错误处理路径**：
[列出文档中记录的所有错误路径]

**副作用**：
[修改的成员变量、触发的信号、I/O 操作]

⚠ 注意事项：[性能警告、线程约束、潜在 Bug 等]
```

**问：某个用户操作的完整流程是什么？**
```
## 场景：[场景名]

**触发入口**：[UI操作/事件]

**完整调用链**：
[直接引用 07_DATA_FLOW.md 中的调用链图，保留线程标注]

**数据变换**：
原始：[类型] → 中间：[类型] → 最终：[类型]

**涉及的状态变化**：
[列出关键成员变量的前后变化]
```

## 阶段 4：主动补充与提示

### 4.1 文档空白检测

在回答过程中，若发现以下情况，**主动向用户说明**：

```
情况A：文档中找不到对应内容
→ 提示："文档未记录此内容，可能是：
   (1) 该部分在文档生成后被新增
   (2) 属于第三方库代码（文档不覆盖外部库）
   建议直接查看源码：cat {PROJECT_ROOT}/src/xxx/xxx.cpp"

情况B：文档标注了 ⚠ 技术债或 Bug
→ 主动说明："注意，文档记录了此处存在已知问题：[具体描述]"

情况C：问题涉及多个文档
→ 说明信息来源："以下回答综合了 05（类定义）和 08（信号槽）两个文档"
```

### 4.2 引导用户高效提问

当用户问题模糊时，根据 `00_AI_CONTEXT.md` 中的核心类列表给出建议：

```
您的问题涉及整个项目，建议缩小范围：
• 如果您想了解 [核心流程]，我可以详细展开 07_DATA_FLOW.md 中的 [场景名]
• 如果您想了解某个具体类，以下是项目的核心类：
  [列出 00_AI_CONTEXT.md 中的核心类速查表]
```

## 特殊场景处理

### 场景A：用户要修改代码

基于文档理解后，帮助用户修改代码时：

1. **先确认影响范围**：在 `02_ARCHITECTURE.md` 中查看该模块的依赖关系
2. **查阅已知问题**：在 `12_KNOWN_ISSUES.md` 中确认是否有相关技术债
3. **理解现有实现**：在 `06_FUNCTION_REFERENCE.md` 中精读目标函数
4. **检查线程安全**：在 `11_THREADING.md` 中确认修改不会引入并发问题
5. **输出修改方案**时，附上："根据文档，此修改还需同步更新 [相关函数/信号/变量]"

### 场景B：用户要排查 Bug

1. 先看 `12_KNOWN_ISSUES.md` —— Bug 可能已被记录
2. 在 `07_DATA_FLOW.md` 中找到对应的用户场景，追踪数据流
3. 在 `11_THREADING.md` 中检查是否是线程安全问题
4. 在 `06_FUNCTION_REFERENCE.md` 中精读嫌疑函数的"错误处理路径"部分

### 场景C：用户要添加新功能

1. 在 `02_ARCHITECTURE.md` 中确定新功能属于哪个模块
2. 在 `05_CLASS_REFERENCE.md` 中找到需要扩展的类
3. 在 `08_QT_SIGNALS_SLOTS.md` 中了解现有信号槽机制，确定接入点
4. 在 `10_MEMORY_LIFECYCLE.md` 中了解对象树结构，确保新对象正确接入

### 场景D：用户询问第三方库用法

文档中仅记录了第三方库**在本项目中的具体用途**。
若用户需要库的通用 API 文档，说明："文档只记录了该库在本项目中的用法，
通用 API 文档请查阅官方文档。"

## 禁止行为

- ❌ 未读 `00_AI_CONTEXT.md` 就直接回答（缺乏项目上下文）
- ❌ 全量加载所有文档（浪费 context，且慢）
- ❌ 在文档有明确记录时，仍去扫描源码（文档已包含更多信息）
- ❌ 对文档中 ⚠ 标注的问题视而不见，不告知用户
- ❌ 在回答中发明文档未记录的"推测性"实现细节

---

## 执行顺序

```
① bash: 检测 _ai_docs/ 是否存在，列出文档清单
② bash: 若存在则检测版本新旧（源码是否比文档更新）
③ bash: 读取 00_AI_CONTEXT.md（必须，建立全局认知）
④ 分析用户问题 → 查映射表 → 确定需要加载的文档
⑤ bash: 精确加载目标文档（对大文档用 grep+sed 定位，禁止全量加载）
⑥ 合成信息 → 按标准模板输出回答
⑦ 检查是否有 ⚠ 警告需要附加说明
⑧ 若有相关的已知问题，附在回答末尾
```
