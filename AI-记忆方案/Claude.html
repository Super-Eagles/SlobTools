01

系统概述与设计原则

核心问题：记忆系统的瓶颈不在于"存哪里"，而在于"检索时能不能找到真正相关的内容"。 关键词搜索只能找到词语重合的内容，而语义向量检索可以理解意思，找到真正相关的历史记忆。

三条设计原则

  • 语义优先 — 向量相似度检索为主，关键词 FTS5 兜底，保证召回率
  • 摘要而非原文 — 存 AI 提炼的摘要，而非原始对话。摘要更短、信息密度更高、检索更准
  • 更新而非只追加 — 记忆有增删改三种操作，避免矛盾信息堆积导致 AI 混乱

两级记忆模型

层级类比人类记忆技术实现生命周期
热记忆工作记忆（当下正在想的）RedisTTL 24h，Session 结束主动固化
冷记忆长期记忆（过去发生过的）SQLite + sqlite-vec永久保存

{

检索模块设计

图 5-1 · 检索完整时序
sequenceDiagram participant U as "用户" participant App as "应用层" participant R as "Redis" participant Emb as "Embedding 服务" participant S as "SQLite" participant AI as "AI" U->>App: "发送问题" App->>R: "KEYS mem:hot:{user_id}:{session_id}:*" R-->>App: "本轮所有热记忆列表" App->>Emb: "问题文本向量化" Emb-->>App: "问题向量 [float...]" App->>S: "vec_search: 向量相似度 Top-8" S-->>App: "相似记忆候选集" App->>S: "FTS5 关键词兜底检索" S-->>App: "关键词匹配记忆" App->>App: "合并去重，取 Top-5" App->>AI: "拼装 Prompt: 记忆 + 问题" AI-->>App: "生成回答" App-->>U: "返回回答"

检索策略详解

两阶段检索：先用向量语义检索召回候选集（Top-8），再与 FTS5 关键词结果合并去重，最终保留 Top-5 注入 Prompt。 两种方式互补：向量能理解语义，FTS5 能精确匹配关键词。
阶段方式数量优先级说明
本轮热记忆Redis 全取全部最高当前 Session 所有记忆，无需检索
向量语义sqlite-vec 余弦相似度Top-8高语义相近的历史记忆，相似度阈值 0.75
关键词兜底FTS5 BM25Top-5中关键词命中，向量结果不足时补充
最终注入合并去重排序Top-5—按相关性分数排序后取前5条

检索核心代码逻辑

// retrieve.js 核心逻辑
async function retrieve(userId, sessionId, question) {

  // 1. 本轮热记忆（全取）
  const hotKeys = await redis.keys(`mem:hot:${userId}:${sessionId}:*`)
  const hotMemories = await Promise.all(hotKeys.map(k => redis.get(k)))
    .then(list => list.map(JSON.parse))

  // 2. 问题向量化
  const queryVec = await embed(question)

  // 3. 向量语义检索（历史冷记忆）
  const vecResults = await db.prepare(`
    SELECT m.*, vec_distance_cosine(v.embedding, ?) AS score
    FROM memories_vec v
    JOIN memories m ON m.rowid = v.rowid
    WHERE m.user_id = ? AND score < 0.25
    ORDER BY score ASC LIMIT 8
  `).all(serializeVec(queryVec), userId)

  // 4. FTS5 关键词兜底
  const keywords = await extractKeywords(question)
  const ftsResults = await db.prepare(`
    SELECT m.* FROM memories m
    JOIN memories_fts f ON m.rowid = f.rowid
    WHERE f.memories_fts MATCH ? AND m.user_id = ?
    ORDER BY rank LIMIT 5
  `).all(keywords.join(' OR '), userId)

  // 5. 合并去重，按分数排序，取 Top-5
  const merged = dedup([...vecResults, ...ftsResults])
    .sort((a, b) => (a.score ?? 0.5) - (b.score ?? 0.5))
    .slice(0, 5)

  return { hot: hotMemories, cold: merged }
}

为什么要有更新/删除机制？ 只追加会导致 AI 同时看到"用户喜欢 A"和三个月后"用户改成喜欢 B"，产生混乱。 更新机制保证每个维度的信息始终保持最新状态。

固化策略

图 7-1 · 两种固化触发方式
flowchart LR subgraph T1["触发方式 A：主动触发（推荐）"] A1["Session 结束信号"] --> B1["读取 Redis 全部热记忆"] B1 --> C1["执行更新决策"] C1 --> D1["批量写入 SQLite"] D1 --> E1["DEL Redis 对应 Keys"] end subgraph T2["触发方式 B：TTL 兜底"] A2["定时任务每 30 分钟"] --> B2["扫描即将过期 Keys"] B2 --> C2["固化到 SQLite"] C2 --> D2["续 TTL 或 DEL"] end

推荐同时使用两种触发方式：方式 A 是主路径，逻辑清晰；方式 B 是兜底，防止因 crash 或异常导致记忆丢失。

固化核心逻辑

// persist.js 核心逻辑
async function persistSession(userId, sessionId) {

  // 1. 取出该 Session 全部热记忆
  const keys = await redis.keys(`mem:hot:${userId}:${sessionId}:*`)
  if (!keys.length) return

  const memories = await Promise.all(keys.map(k => redis.get(k)))
    .then(list => list.map(JSON.parse))

  // 2. 按轮次排序
  memories.sort((a, b) => a.turn - b.turn)

  // 3. 逐条执行更新决策后写入
  for (const mem of memories) {
    const similar = await findSimilarInSQLite(userId, mem.embedding, 0.85)
    if (!similar) {
      await insertMemory(mem)
    } else {
      const isConflict = await checkConflict(mem.summary, similar.summary)
      if (isConflict) {
        await deleteMemory(similar.id)
        await insertMemory(mem)
      } else {
        await updateMemory(similar.id, mergeMemory(similar, mem))
      }
    }
  }

  // 4. 清除 Redis 热记忆
  await redis.del(...keys)
  await updateSessionStatus(sessionId, 'ended')
}

Prompt 注入设计

注入格式需要让 AI 明白：这些是背景记忆，而非本次对话内容。

// inject.js 生成的 system prompt 片段
function buildSystemPrompt(hotMemories, coldMemories) {
  const hot = hotMemories.map((m, i) =>
    `[本轮记忆-${i+1}] ${m.summary}`
  ).join('\n')

  const cold = coldMemories.map((m) =>
    `[历史记忆 ${m.created_at.slice(0,10)}] ${m.summary}`
  ).join('\n')

  return `
你拥有以下长期记忆，这些是与用户过去交流中积累的重要信息，
请在回答时自然地结合这些记忆，不要明确说"根据记忆"：

${cold || '（暂无相关历史记忆）'}

本轮对话摘要：
${hot || '（本轮首次对话）'}

---
请基于以上背景信息回答用户问题。
`.trim()
}

注入量控制原则

记忆类型最大条数截断策略
本轮热记忆全部（通常 <20 轮）超过 20 轮时取最近 10 条
历史冷记忆5 条按相似度分数排序，只取 Top-5
总 Token 预算~1200 tokens超出时优先保留热记忆，冷记忆继续截断

向量化方案

方案模型维度适用场景成本
OpenAI（推荐）text-embedding-3-small1536生产环境，效果最好$0.02/1M tokens
本地（推荐备选）nomic-embed-text via Ollama768无网络依赖，数据不出本机免费
国内替代通义千问 text-embedding-v21536国内网络环境按量计费
建议：封装 embed(text) 函数，内部切换模型。 开发阶段用 Ollama 本地模型，上线后换 OpenAI，代码无需改动。 
// embedding.js
async function embed(text) {
  if (process.env.EMBED_PROVIDER === 'openai') {
    const res = await openai.embeddings.create({
      model: 'text-embedding-3-small',
      input: text
    })
    return res.data[0].embedding
  } else {
    // Ollama 本地模型
    const res = await fetch('http://localhost:11434/api/embeddings', {
      method: 'POST',
      body: JSON.stringify({ model: 'nomic-embed-text', prompt: text })
    })
    return (await res.json()).embedding
  }
}
