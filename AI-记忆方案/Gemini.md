一份真正“可上线”的记忆系统，不能只是简单的向量数据库堆砌。它必须解决**检索噪声、上下文爆炸**和**长期衰减**的问题。

以下是我为你设计的架构，采用分层存储逻辑，兼顾了 C++ 开发者关注的性能与 AI Agent 要求的语义深度。

---

## 记忆系统架构：三层金字塔模型

### 1. 感知层：工作记忆 (Working Memory)
* **定位**：当前对话的原始上下文。
* **存储**：Redis 或内存循环队列（Sliding Window）。
* **逻辑**：保留最近 $N$ 轮对话。这是 AI 反应最敏锐的部分。
* **上线要点**：必须限制 Token 数量，防止请求超时。

### 2. 演化层：片段记忆 (Episodic Memory)
* **定位**：过去发生的具体事件或任务流。
* **存储**：向量数据库 (Qdrant / Milvus) + 关系型数据库 (SQLite/PostgreSQL)。
* **逻辑**：
    * **提取**：将对话摘要化（Summary），提取出“用户遇到了 MSVC 编译错误”这类事实。
    * **向量化**：对摘要进行 Embedding。
    * **检索**：通过 **Vector Search + 时间权重** 进行双路召回。
* **上线要点**：异步写入。不要让 Embedding 过程阻塞主线程。

### 3. 核心层：语义记忆 (Semantic Memory)
* **定位**：用户画像、偏好、固定事实（如：Fiann 是 C++ 程序员，在新加坡）。
* **存储**：结构化数据库 (SQLite) 或 知识图谱 (Neo4j)。
* **逻辑**：
    * 从对话中识别实体（Entity Extraction）。
    * **增量更新**：如果用户说“我不喜欢老女人”，直接覆盖旧标签，而不是存两条冲突记录。
* **上线要点**：这部分数据最精准，检索优先级最高。

## 核心工作流程 (Pipeline)

| 阶段 | 动作 | 技术细节 |
| :--- | :--- | :--- |
| **输入 (In)** | 意图解析 | 提取关键词、时间戳、地理位置。 |
| **检索 (Retrieve)** | 多路召回 | 1. 查 SQLite 里的用户偏好；2. 查向量库里的相似经历。 |
| **压缩 (Compress)** | 冲突检测 | 如果检索到两条矛盾记忆，让 LLM 判断哪条是新的，剔除旧的。 |
| **生成 (Gen)** | 上下文注入 | 将“画像 + 相关往事 + 当前对话”拼成最终 Prompt。 |
| **归档 (Arch)** | 异步总结 | 对话结束后，后台触发脚本，将本次对话压缩存入 LTM。 |

## 生产环境的“避坑”建议

1.  **忘记机制 (Forgetting Curve)**：
    * 不要存储每一句话。对于超过 30 天且从未被检索过的向量，应当降低其权重或物理删除。
2.  **隐私脱敏**：
    * 在存入向量库前，正则过滤掉密码、密钥等敏感信息。
3.  **高性能检索 (C++ 视角)**：
    * 如果你在 Windows 下开发，推荐使用 **SQLite** 存储结构化画像（就像你现在做的那样），因为它足够轻量且单机性能极佳。
    * 向量库建议部署在 Docker 容器中，通过 gRPC 与你的 C++ 主程序通信。
